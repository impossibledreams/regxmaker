0. Introduction.

          Thank you using RegXmaker. This utility is designed to help people with using regular-expressions (sometimes called "regex" or RE). The tool is not designed to do all the work for you, rather it helps you generate regex and then you can modify them for your needs. The initial construction gives you something to work with rather than starting from scratch. RegXmaker compiles regex based on strings, URLs and files that you provide. It compares them and generates a regex that will match data in those sources. RegXmaker is able to process information from files, user-entered strings and URLs. There are also several options available to specify how accurate the regex should be, or how closely the regex matches. These options are described later on in this document. Please note that due to many factors RegXmaker is not able to load files and URLs larger than 8 megabytes. Due to the processing speed of RegXmaker it is not recommended to process files and URLs larger than half of a megabyte in size because this may require a long period of time.
          If you have any questions or comments please feel free to visit our website at www.solidmatrix.com or email us at research@solidmatrix.com.

0.1. About Regular Expressions.

          Regular expressions are like a language designed to describe patterns of characters in text. They are used to identify, extract or substitute specific strings within a larger body of text. You can ask questions such as "Does this string match the pattern?" and also use regex to modify strings, or to split strings apart in various ways. A common use is to extract email addresses or links from HTML documents. This can be done much faster when using regular expressions than writing the code from scratch. Regular expressions are an essential part of many scripting languages used on the Web today such as Perl, Python, PHP and others. It is also used extensively within the UNIX world. Regex modules are available for virtually any language and platform ranging from UNIX-based shell scripts to Microsoft's Visual Basic. RegXmaker closely mimics the syntax of regular expressions as they are used in Perl (visit www.perl.com for more information).
          A regular expression is a pattern of logical notation that represents a set of possible strings. For example given two strings "Today is Monday" and "Yesterday was Sunday" a simple regular expression can be produced that matched either string ("/Today is .*n.*day/"). This string is deciphered as follows: "/" indicates start and end of regex, "Today is " is section of the text that must be present, ".*" indicates that any characters in any number can be present followed by letter "n", another set of any characters followed by the word "day". This regex is not very exact so it can match things like "Today is any day" or "Today is a nice day". To get more exact results you must be precise when writing REs.

0.2. Overall Layout.

          The program window is split into several sections. On the top is the menu accessible by pressing F10 (not ALT like in Windows). Below the menu is a horizontal toolbar contain commands from the File and Edit menus. The program screen is split into two sections: the left white section contains the list of data sources that are being used to generate the regex and the right blue section contains the results windows. On the left side of the list section is a vertical toolbar with icons corresponding to the List menu. These commands are used to manage data sources in the list. All icons on the toolbars have a description that can be displayed by putting the mouse arrow on top of the icon (not clicking). Toolbars can be removed by using the Customize menu and can float if you click and drag the toolbar by clicking on their left edge. To dock floating toolbars you must close them by clicking on the close button on the top right edge. Sometimes the vertical list toolbar will dock horizontally instead. This is normal behavior.

4.1. Compile Level.           

          These levels define how specific resulting regex will be.
Deep Compile
Will list all possible cases from the list of sources. Example: if data sources are "abc" and "def" the resulting regex is /[ad][be][df]/.
 
Semi-Deep Compile
The resulting regular expression will contain range of highest to lowest example cases between the static sections of the given strings. Example: for strings "a0" and "f4" the resulting regex is /[a-f][0-4]/.
 
Surface Compile
The resulting regular expression will contain only types of cases between the static sections of the given strings. Example: for strings "a1B" and "c2D" the resulting regex is /[a-z][0-9][A-Z]/.
 
No Compile
Will only show static sections with dynamic ones shown as ".*". Example: for strings "1abcdefg2" and "1hijklmnop2" the resulting regular expression is /a.*g/.
Back to top

4.2. Accuracy Level.           

          This specifies what is considered a match between data sources.           The choices are:
Letter Accuracy
A single letter is considered a match.
 
Word Accuracy
Word accuracy is based on the average word size of 5 letters.
 
Line Accuracy
Line accuracy is based on standard line size of 80 letters.
          Accuracy levels are based on letter counts instead of the language concept of "word" and "line" in order to allow different accuracy levels for binary files.

Back to top

4.3. Text Format.           

          These define different encodings used. Available encodings are:
US-ASCII
Seven-bit ASCII, a.k.a. ISO646-US, a.k.a. the Basic Latin block of the Unicode character set
ISO-8859-1
ISO Latin Alphabet No. 1, a.k.a. ISO-LATIN-1
UTF-8
Eight-bit Unicode Transformation Format
UTF-16BE
Sixteen-bit Unicode Transformation Format, big-endian byte order
UTF-16LE
Sixteen-bit Unicode Transformation Format, little-endian byte order
UTF-16
Sixteen-bit Unicode Transformation Format, byte order specified by a mandatory initial byte-order mark (either order accepted on input, big-endian used on output)
          This is an advanced option. Do not change anything unless you know what you are doing. Refer to sections above to find out how this setting affects loading local and remote files as well as writing output. For more information see www.unicode.org.
